# CallNexus 功能路线图

本文档用于统一记录 CallNexus 呼叫中心系统的功能模块、开发顺序和当前进度。后续 AI 或 Claude Code 接手开发时，先阅读本文档，再结合 `CALL_CENTER_ENGINEERING_GUIDE.md` 和 `AI_DEVELOPMENT_GUIDE.md` 执行具体实现。

## 状态说明

- `[DONE]`：基础功能已实现，可以进入联调或后续增强。
- `[PARTIAL]`：已有代码或页面，但能力未闭环，仍需要继续开发。
- `[PLANNED]`：规划中，尚未正式实现。
- `[BLOCKED]`：代码侧已有准备，但依赖服务器、FreeSWITCH 或外部配置。

## 当前进度

当前处于 **阶段 5：声音媒体管理与 IVR 第一版**：

- 阶段 1：呼叫中心基础底座已经基本完成，包括 FreeSWITCH 节点、SIP 账号、坐席、坐席工作台、ESL 控制、客户、工单、动态表单模板。
- 阶段 2：FreeSWITCH 动态 Directory 已完成联调，后台新增 SIP 账号后，FreeSWITCH 可以通过 CallNexus 动态获取分机目录并完成注册。
- 阶段 3：网关、号码管理、动态 Gateway XML 和动态 Dialplan 第一版已经完成，并已通过真实外线呼入验证。
- 阶段 4：CDR 业务通话聚合、录音自动触发、MinIO 分类存储、在线试听、客户和工单显式关联已经完成第一版。
- 阶段 5：声音媒体发布同步和 IVR 拖拽设计器第一版已完成，等待真实呼入联调。

当前最优先事项：

1. 执行 `V18`，创建并发布第一条 IVR 流程。
2. 将真实 DID 绑定到已发布 IVR，联调提示音、DTMF 分流和转接分机。
3. 增加 IVR 版本历史、预览和回滚。
4. IVR 联调稳定后进入技能组和队列。

## 推荐开发顺序

### 阶段 0：基础平台

状态：`[DONE]`

目标：保证系统可以登录、授权、菜单展示、基础租户体系可用。

已完成内容：

- 后台登录。
- 系统菜单基础能力。
- 首页从介绍页调整为呼叫中心运营首页。

### 阶段 1：呼叫中心基础底座

状态：`[PARTIAL]`

目标：先让坐席、SIP 分机、FreeSWITCH 节点、ESL 控制和基础客户工单跑通。

已完成内容：

- FreeSWITCH 节点管理。
- SIP 账号管理。
- 坐席管理。
- 坐席电话条。
- ESL 命令控制。
- ESL 事件监听基础结构。
- 客户管理。
- 工单管理。
- 动态表单模板。
- 客户跟进记录。

仍需补齐：

- ESL 事件落库。
- 通话生命周期状态强一致处理。
- 软电话挂断后浏览器端状态自动同步的更多边界场景。
- 呼叫保持、转接、咨询、三方通话等坐席高级操作。

### 阶段 2：FreeSWITCH 动态配置

状态：`[PARTIAL]`

目标：后台新增或修改 SIP 账号、网关、路由后，FreeSWITCH 不需要手工改 XML 文件。

当前已完成：

- 后端动态 Directory XML 接口。
- Directory 安全 Token 配置。
- SIP 账号查询和 XML 渲染结构。
- 服务器侧 `mod_xml_curl` 配置和联调。
- 后台新增 SIP 账号后，普通 SIP 软电话可以注册成功。

后续需要开发：

- 动态 Dialplan XML。
- 动态 Gateway XML 后端输出已完成，等待服务器侧 FreeSWITCH 配置联调。
- 动态 ACL XML。
- XML 请求日志和调试页面。

### 阶段 3：网关、号码和呼入路由

状态：`[PARTIAL]`

目标：让系统可以接运营商线路，支持外线呼入。

开发内容：

- 网关管理：SIP Trunk 地址、端口、认证、方向、状态。第一版和动态 Gateway XML 后端输出已完成，尚未完成服务器侧联调。
- 号码管理：DID、主叫号码、归属租户、启停状态。
- 呼入路由：号码绑定 IVR、队列、坐席、技能组。
- 呼出路由：按租户、坐席、号码池选择出口网关。

### 阶段 4：通话记录和录音

状态：`[PARTIAL]`

目标：每通电话都能追踪、查询、回放和关联业务。

开发内容：

- CDR 通话记录表。
- ESL 事件转通话生命周期。
- 业务通话、底层通话腿和操作时间线聚合。
- 录音文件路径、时长、存储状态。
- 客户详情和工单详情增加通话记录 Tab。
- 呼叫失败原因归类。

### 阶段 5：IVR

状态：`[PLANNED]`

目标：支持客户来电后按语音菜单自动分流。

开发内容：

- IVR 流程管理。
- IVR 节点管理。
- 语音文件管理。
- DTMF 按键路由。
- IVR 执行记录。
- IVR 版本发布。

### 阶段 6：技能组和队列

状态：`[PLANNED]`

目标：支持标准呼叫中心的排队和分配。

开发内容：

- 技能组管理。
- 坐席技能绑定。
- 队列管理。
- 队列策略：最长空闲、轮询、随机、优先级。
- 等待音、超时、溢出、无人接听处理。
- 坐席状态和队列状态实时看板。

### 阶段 7：外呼任务

状态：`[PLANNED]`

目标：支持人工和自动化外呼。

建议顺序：

1. 预览式外呼：坐席确认客户后手动拨打。
2. 渐进式外呼：系统按空闲坐席自动发起呼叫。
3. 预测式外呼：系统根据接通率和坐席空闲率预测拨号。
4. 机器人外呼：AI 或语音机器人自动完成外呼。

开发内容：

- 外呼任务。
- 外呼名单。
- 外呼批次。
- 号码去重和黑名单。
- 外呼结果。
- 失败重试。
- 外呼统计。

### 阶段 8：AI 和知识库

状态：`[PLANNED]`

目标：让人工接听时获得实时辅助，而不是一开始就替代人工。

第一阶段先做人工辅助：

- 实时语音转写。
- 前端展示双方对话流。
- 根据知识库推荐话术。
- 自动识别客户意图。
- 自动提取客户字段。
- 自动生成工单草稿。
- 自动生成通话摘要。

第二阶段再做半自动处理：

- AI 推荐客户分类。
- AI 推荐工单类型和优先级。
- AI 推荐下一步跟进动作。
- 人工确认后保存。

第三阶段再考虑机器人：

- 机器人外呼。
- 机器人 IVR。
- 人机协同转人工。

### 阶段 9：报表和监控

状态：`[PLANNED]`

目标：支撑运营管理和系统稳定性。

开发内容：

- 呼入量、呼出量、接通率。
- 坐席在线时长、通话时长、空闲时长、示忙时长。
- 队列等待时长、放弃量、呼损率。
- 网关状态、ESL 状态、FreeSWITCH 状态。
- 告警中心。
- 导出报表。

## 当前下一步建议

下一步不要直接做 AI、机器人或预测外呼，当前基础链路还没完全闭环。

建议立刻处理：

1. 执行 `V14__aggregate_call_sessions.sql` 和 `V15__add_call_recording_and_business_links.sql`。
2. 按 `FREESWITCH_RECORDING_DEPLOYMENT.md` 部署录音上传脚本并联调 MinIO。
3. 验证真实呼入、呼出、内部分机通话的聚合、录音回放和客户/工单显式关联。
4. 联调完成后开发 IVR、技能组和队列。

完成以上内容后，再进入 IVR 和队列会更稳，因为 IVR、队列、外呼和 AI 都依赖稳定的号码、路由、通话记录和事件模型。
